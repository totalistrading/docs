**Architecture Update (v2.1)**: The system uses persistent participant vaults.
Positions are created atomically in a single transaction.

**Key architecture points:**
- Single atomic TX per trade via `create_position`
- Persistent vault PDA per participant: `["vault", owner_wallet]`
- Position PDA per trade: `["position", position_id]`
- MM collateral calculated via ILP (`mm_collateral_locked` may be less than `mm_risk`)
- Position statuses: `pending` → `active` → `settled_win`/`settled_loss`/`cancelled`

**Other key changes from current backend:**
- Consistent `{ data, meta }` response envelope
- `venue` field on legs, markets, and events (multi-venue from day 1)
- Auth header standardized to `X-API-Key` (was `TOTALIS_API_KEY`)
- Wire format is **snake_case** (Go/JSON convention). The frontend API client
  transforms to **camelCase** at the boundary (`bet_amount` → `betAmount`).
  Frontend types in `src/types/` are all camelCase.
- All pagination is cursor-based (leaderboard and MM history migrated from offset)
- Settlement outcomes use `winner: "user" | "mm"` everywhere (unified `WinnerSide`)
- `success: boolean` removed from responses where errors throw (redeem-invite, waitlist)
- Error responses include `retry_after` for rate-limited requests
- `kalshi_url` renamed to `venue_url` on markets
- State-transition endpoints use `POST` not `PUT` or `DELETE`
- Pagination cursors encode `(created_at, id)` sort key, not offset

## Error Codes

| HTTP | Code | Meaning |
|------|------|---------|
| 400 | `VALIDATION_ERROR` | Invalid request body/params |
| 401 | `UNAUTHORIZED` | Missing or invalid auth |
| 403 | `FORBIDDEN` | Authenticated but not allowed (geo-block, not approved) |
| 404 | `NOT_FOUND` | Resource doesn't exist |
| 409 | `CONFLICT` | Duplicate action (e.g., double-quote) |
| 429 | `RATE_LIMITED` | Too many requests - `retry_after` field set (seconds) |
| 413 | `PAYLOAD_TOO_LARGE` | Request body exceeds 256KB limit |
| 500 | `INTERNAL_ERROR` | Server error |

## Rate Limits

- Anonymous: 100 req/min per IP
- Authenticated: 300 req/min per (IP + user)
- Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- Request body limit: 256KB

## Idempotency

Mutation endpoints (`POST /rfqs`, `POST /mm/quotes`) accept an optional `Idempotency-Key`
header. If the server has already processed a request with the same key (within a 24h TTL
window), it returns the cached response without re-executing.

```
Idempotency-Key: <client-generated UUID>
```

### Market Maker (Auth Required)

#### `POST /mm/quotes/:quoteId/confirm`
Confirm accepted quote, triggers vault position creation.

When MM confirms:
1. A `vault_position` record is created with status `pending`
2. The vault-job-service processes it asynchronously
3. On-chain position is created atomically (single TX with both deposits)
4. Status transitions to `active` on success

**Balance handling:**
- No balance pre-check at confirm time - confirmation always succeeds if quote is valid
- vault-job-service fetches on-chain vault state and auto-deposits the shortfall from the user's/MM's ATA if needed
- If ATA has insufficient USDC for the required deposit, the TX fails and `retry_count` increments
- After `MAX_RETRIES` (default 5, configurable via `VAULT_JOB_MAX_RETRIES`) exhausted, position transitions to `error` status

**`mm_collateral_locked` calculation (ILP):**
- Calculated at vault-job-service processing time, NOT at confirm time
- Inputs: MM's existing active positions (mm_risk + legs) + the new position
- Formula: `mm_collateral_locked = worst_case(existing + new) - worst_case(existing)`
- This enables collateral netting when positions hedge each other

**Response:**
```json
{
  "data": {
    "status": "confirmed",
    "quote_id": "quote-uuid",
    "execution_id": "exec-uuid"
  }
}
```

## State Transitions

### RFQ Lifecycle

```
open → quoted       (first quote received)
open → cancelled    (user cancels)
open → expired      (expires_at reached)
quoted → accepted   (user accepts a quote)
quoted → cancelled  (user cancels)
quoted → expired    (expires_at reached)
accepted → confirmed (MM confirms the accepted quote)
accepted → expired  (MM fails to confirm before deadline)
confirmed → executed (vault position created on-chain)
executed → settled  (all legs resolved, payout distributed)
executed → cancelled (timeout or manual admin cancellation)
```

### Quote Lifecycle

```
pending → accepted   (user accepts this quote)
pending → rejected   (user rejects this quote)
pending → withdrawn  (MM withdraws)
pending → expired    (RFQ expires or quote valid_until reached)
accepted → confirmed (MM confirms)
accepted → expired   (MM fails to confirm before deadline)
confirmed → executed (position funded)
```

### Vault Position Lifecycle (NEW)

```
pending → active         (on-chain position created successfully)
pending → [retry]        (creation failed, retry_count < MAX_RETRIES, stays pending and retries)
pending → error          (retry_count >= MAX_RETRIES, exhausted)
active → settling        (settlement in progress, claimed by worker)
active → cancelling      (cancellation in progress)
settling → settled_win   (user won, payout distributed)
settling → settled_loss  (MM won, payout distributed)
settling → active        (settlement failed, reverted for retry)
cancelling → cancelled   (position cancelled, funds returned)
cancelling → active      (cancellation failed, reverted for retry)
```

**Key points:**
- Position creation is atomic (single TX)
- Intermediate `settling`/`cancelling` states for async processing
- `error` is a terminal state requiring manual intervention

**Retry behavior:**
- vault-job-service retries failed positions up to `MAX_RETRIES` (default 5, configurable via `VAULT_JOB_MAX_RETRIES` env var)
- On each failure, `error` field is set and `retry_count` increments
- When `retry_count >= MAX_RETRIES`: position transitions to `error` status, `position:error` WebSocket event emitted
- No automatic timeout - settlement only happens when underlying markets resolve (via Kalshi/Polymarket)

### Reconnection

Client implements exponential backoff (1s → 2s → 4s → ... → 30s cap). On reconnect:
1. Re-authenticate
2. Re-subscribe to all channels
3. Send `resync` for each channel with `last_seq`
4. Full HTTP refetch as safety net

## Migration Notes

### Escrow → Vault Architecture Changes

The vault-based system replaces per-trade escrows with persistent participant vaults. Key differences:

| Aspect | Escrow Flow (v1) | Vault Flow (v2.1) |
|--------|------------------|-------------------|
| Account creation | 3-step: create → user_fund → mm_fund | Single atomic TX via `create_position` |
| PDA seeds | `["escrow", parlay_id]` | Vault: `["vault", owner_wallet]`, Position: `["position", position_id]` |
| Collateral | Full amount from both parties | User: full stake, MM: ILP-calculated (`mm_collateral_locked` ≤ `mm_risk`) |
| Funding model | Per-trade escrow account | Persistent vault with locked collateral |
| Status progression | created → user_funded → active → settled | pending → active → settled_win/settled_loss |

### Vault Initialization

Vaults are created automatically when a user's first position is created. There is no separate "deposit before trading" step - the vault is initialized as part of the first `create_position` transaction. Users can deposit additional funds to their vault after it exists.

### WebSocket Event Migration

| Escrow Event (v1) | Position Event (v2.1) | Notes |
|-------------------|----------------------|-------|
| `escrow:created` | `position:created` | Emitted after atomic position creation (was after escrow PDA created) |
| `escrow:user_funded` | (removed) | Not applicable - vault flow is atomic |
| `escrow:active` | (removed) | Position is `active` immediately after creation |
| `escrow:settled` | `position:settled` | Same semantics |
| `escrow:cancelled` | `position:cancelled` | Same semantics |
| `escrow:failed` | `position:failed` | Same semantics |
| `escrow:error` | `position:error` | Same semantics |

### Escrow → Vault Field Mapping

| Escrow Field | Vault Field | Notes |
|--------------|-------------|-------|
| `escrow_pda` | `position_pda` | Position PDA, not vault PDA |
| `escrow_token_account` | `vault_token_account` | Token account associated with vault |
| `user_deposit` | `user_stake` | User's bet amount |
| `mm_deposit` | `mm_risk` | Max payout if user wins |
| - | `mm_collateral_locked` | Actual collateral locked via ILP (≤ mm_risk) |
| `parlay_id_onchain` | `position_id` | 16-byte UUID seed for position PDA |
| `escrow_status` | `status` | See status mapping below |
| `deadline` | - | Not used in vault flow (settlement is event-driven) |

### Status Mapping

| Escrow Status | Vault Position Status | Notes |
|--------------|----------------------|-------|
| `created` | `pending` | Awaiting on-chain creation |
| `user_funded` | - | Not applicable (atomic flow) |
| `active` | `active` | Position live on-chain |
| `settling` | `settling` | Settlement in progress |
| `settled` | `settled_win` / `settled_loss` | Split by outcome |
| `cancelling` | `cancelling` | Cancellation in progress |
| `cancelled` | `cancelled` | Position cancelled |
| `stale` | - | Handled via retry logic |
