---
**Architecture Update (v2.1)**: Added `vaults` and `vault_positions` tables for the
vault-based position system. The escrow-related fields on `parlay_tickets` are retained
for backward compatibility but are deprecated in favor of vault positions.

---

## Design Decisions


| Decision                                         | Rationale                                                                                                                                                                                                                                          |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Privy DID as `users.id` PK**                   | The DID *is* the identity - no value in an extra UUID + `privy_id` column                                                                                                                                                                          |
| **`api_keys.entity_type` not exposed in API**   | DB has `entity_type` (always `'unified'`, CHECK enforced) + `entity_id` (Privy DID). API exposes only `entity_id` - no branching logic for entity_type                                                                                             |
| **Separate `vaults` and `vault_positions` tables** | Vaults are persistent per-participant; positions are per-trade. Clean separation enables efficient queries for balance, locked collateral, and position history                                                                                    |
| **`vault_positions.position_id` is a 16-byte UUID** | Matches on-chain position PDA seed; stored as BYTEA for exact byte-level matching                                                                                                                                                                  |
| **Position status includes intermediate states** | `pending`, `settling`, `cancelling` enable async job processing with proper claim/release semantics                                                                                                                                                |
| `**leg_prices` as JSONB on `quotes`**            | Quotes have 2–6 legs (bounded), are immutable post-creation, and are always read whole. JSONB avoids N+1 writes, Postgres jsonpath operators cover any analytical query                                                                           |
| `**rfq_events` is a generic audit log**          | Tracks status transitions for tickets, quotes, and positions in one place with `entity_type`/`entity_id`                                                                                                                                          |
| `**user_stats` stores terminal aggregates only** | Per-status counters are noise - active statuses are a tiny live query per user (bounded). Terminal aggregates (wins, losses, pnl) are pre-computed and indexed for leaderboard                                                                    |
| `**venue` required everywhere**                  | Multi-venue is a first-class requirement - `venue` is `NOT NULL` on all market, leg, and result tables                                                                                                                                             |
| **Cursor pagination uses sort-key encoding**     | Cursors encode `(created_at, id)` - not offset. Offset pagination breaks under concurrent inserts at scale                                                                                                                                         |

## Schema: `rfq`

All tables live in the `rfq` schema.

```sql
CREATE SCHEMA IF NOT EXISTS rfq;

-- ============================================================
-- INVITE CODES
-- (Defined first - users.invite_code_id references it)
-- ============================================================
CREATE TABLE rfq.invite_codes (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    code         VARCHAR(6) NOT NULL UNIQUE
                 CHECK (code ~ '^[A-Z]{6}$'),
    source       TEXT NOT NULL DEFAULT 'admin',     -- 'admin' or referrer user_id
    source_label TEXT,                               -- campaign label
    redeemed_by  TEXT,                               -- user_id or NULL
    redeemed_at  TIMESTAMPTZ,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_invite_codes_source ON rfq.invite_codes (source);

-- ============================================================
-- USERS
-- ============================================================
CREATE TABLE rfq.users (
    id                    TEXT PRIMARY KEY,          -- Privy DID ("did:privy:abc123")
    username              TEXT NOT NULL
                          CHECK (
                              username ~ '^0x[a-fA-F0-9]{40}$'
                              OR username ~ '^[a-zA-Z0-9_]{3,20}$'
                          ),
    email                 TEXT,
    wallet_address        TEXT,                      -- Solana pubkey (Privy embedded wallet)
    approved              BOOLEAN NOT NULL DEFAULT FALSE,
    invite_code_id        UUID REFERENCES rfq.invite_codes(id),
    total_usdc_airdropped NUMERIC(20, 6) NOT NULL DEFAULT 0,
    created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Case-insensitive uniqueness on username
CREATE UNIQUE INDEX users_username_lower_idx ON rfq.users (lower(username));
CREATE INDEX idx_users_wallet ON rfq.users (wallet_address);

-- ============================================================
-- API KEYS
-- Uses entity_type/entity_id pattern (entity_type always 'unified')
-- ============================================================
CREATE TABLE rfq.api_keys (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    key_prefix    VARCHAR(12) NOT NULL UNIQUE,       -- e.g., "api_live_xxxx" - safe to log
    key_hash      VARCHAR(64) NOT NULL UNIQUE,        -- bcrypt hash of full key
    entity_type   VARCHAR(20) NOT NULL
                  CHECK (entity_type = 'unified'),    -- Always 'unified'
    entity_id     VARCHAR(100) NOT NULL,              -- Privy DID (user's id)
    name          VARCHAR(255),                        -- human label
    status        VARCHAR(20) NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active', 'revoked', 'expired')),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at    TIMESTAMPTZ,
    last_used_at  TIMESTAMPTZ,
    revoked_at    TIMESTAMPTZ,
    revoked_reason VARCHAR(255)
);

CREATE INDEX idx_api_keys_entity ON rfq.api_keys (entity_type, entity_id);
CREATE INDEX idx_api_keys_hash ON rfq.api_keys (key_hash);
CREATE INDEX idx_api_keys_active ON rfq.api_keys (status)

    user_id              TEXT NOT NULL REFERENCES rfq.users(id),
    status               TEXT NOT NULL DEFAULT 'open'
                         CHECK (status IN (
                             'open', 'quoted', 'accepted', 'confirmed',
                             'executed', 'cancelled', 'expired', 'settled'

    bet_amount           NUMERIC(15, 2) NOT NULL CHECK (bet_amount >= 1 AND bet_amount <= 5000),
    implied_probability  NUMERIC(10, 6),

    deposit_user_tx      VARCHAR(128),
    deposit_mm_tx        VARCHAR(128),
    settle_tx            VARCHAR(128),
    close_tx             VARCHAR(128),

    -- Timestamps

-- ============================================================
-- VAULTS (NEW - Persistent per-participant vault accounts)
-- Derived from: rfq-apis/src/lib/db/vaults.ts
-- On-chain: parlay-vaults/programs/parlay_vaults/src/state.rs (Vault struct)
-- Seeds: ["vault", owner_wallet]
--
-- UNIQUE(owner_id, owner_type) rationale:
--   On-chain: One vault per wallet (seeds = ["vault", owner_wallet]).
--   DB: Tracks user/mm roles separately. If the same person acts as both
--   user and MM (same wallet), they get two DB rows with different owner_type
--   values, but vault_pda UNIQUE ensures no on-chain conflicts since both
--   rows point to the same PDA.
--
-- Balance caching:
--   gross_balance and locked_collateral are cached copies of on-chain state.
--   They are updated atomically alongside position state changes (create,
--   settle, cancel) to avoid live RPC calls on read paths.
-- ============================================================
CREATE TABLE rfq.vaults (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id            TEXT NOT NULL,                      -- Privy DID
    owner_type          VARCHAR(10) NOT NULL
                        CHECK (owner_type IN ('user', 'mm')),
    owner_wallet        VARCHAR(44) NOT NULL,               -- Solana wallet address
    vault_pda           VARCHAR(44) NOT NULL UNIQUE,        -- On-chain vault PDA
    vault_token_account VARCHAR(44) NOT NULL,               -- Associated USDC token account
    init_tx             VARCHAR(128) NOT NULL,              -- TX that initialized vault

    -- Cached on-chain balances (updated atomically with position state changes)
    gross_balance       NUMERIC(20, 6) NOT NULL DEFAULT 0
                        CHECK (gross_balance >= 0),
    locked_collateral   NUMERIC(20, 6) NOT NULL DEFAULT 0
                        CHECK (locked_collateral >= 0 AND locked_collateral <= gross_balance),
    free_balance        NUMERIC(20, 6) GENERATED ALWAYS AS (gross_balance - locked_collateral) STORED,

    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE (owner_id, owner_type)
);

CREATE INDEX idx_vaults_owner_wallet ON rfq.vaults (owner_wallet);
CREATE INDEX idx_vaults_pda          ON rfq.vaults (vault_pda);

-- ============================================================
-- VAULT POSITIONS (NEW - Per-trade positions within vaults)
-- Derived from: rfq-apis/src/lib/db/vault-positions.ts
-- On-chain: parlay-vaults/programs/parlay_vaults/src/state.rs (ParlayPosition struct)
-- Seeds: ["position", position_id]
-- ============================================================
CREATE TABLE rfq.vault_positions (
    id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ticket_id            UUID NOT NULL REFERENCES rfq.parlay_tickets(id) ON DELETE CASCADE,
    position_id          BYTEA NOT NULL UNIQUE
                         CHECK (octet_length(position_id) = 16),  -- 16-byte UUID, matches on-chain seed
    position_pda         VARCHAR(44),                       -- On-chain position PDA (set after creation)
    user_vault_pda       VARCHAR(44) REFERENCES rfq.vaults(vault_pda),  -- User's vault PDA
    mm_vault_pda         VARCHAR(44) REFERENCES rfq.vaults(vault_pda),  -- MM's vault PDA
    user_wallet          VARCHAR(44),                       -- User's wallet address
    mm_wallet            VARCHAR(44),                       -- MM's wallet address
    user_stake           NUMERIC(15, 2) NOT NULL,           -- User's bet amount (was user_deposit)
    mm_risk              NUMERIC(15, 2) NOT NULL,           -- Max payout owed to user if they win
    mm_collateral_locked NUMERIC(15, 2)
                         CHECK (mm_collateral_locked >= 0 AND mm_collateral_locked <= mm_risk),
    total_payout         NUMERIC(15, 2) NOT NULL,           -- user_stake + mm_risk
    status               VARCHAR(20) NOT NULL DEFAULT 'pending'
                         CHECK (status IN (
                             'pending',       -- Created in DB, awaiting on-chain creation
                             'active',        -- On-chain position created and funded
                             'settling',      -- Settlement in progress (claimed by worker)
                             'cancelling',    -- Cancellation in progress
                             'settled_win',   -- User won, payout distributed
                             'settled_loss',  -- MM won, payout distributed
                             'cancelled',     -- Position cancelled, funds returned
                             'error'          -- Exhausted retries (retry_count >= MAX_RETRIES)
                         )),
    create_position_tx   VARCHAR(128),                      -- TX that created on-chain position
    settle_tx            VARCHAR(128),                      -- Settlement TX
    cancel_tx            VARCHAR(128),                      -- Cancellation TX
    close_tx             VARCHAR(128),                      -- Position close TX (if separate)
    error                TEXT,                              -- Last error message
    retry_count          INTEGER NOT NULL DEFAULT 0,        -- Number of retry attempts
    created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    settled_at           TIMESTAMPTZ,
    updated_at           TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Find pending positions for job worker
CREATE INDEX idx_vault_positions_pending ON rfq.vault_positions (created_at ASC)
    WHERE status = 'pending';

-- Find active positions for settlement
CREATE INDEX idx_vault_positions_active ON rfq.vault_positions (created_at ASC)
    WHERE status = 'active';

-- Find positions by ticket
CREATE INDEX idx_vault_positions_ticket ON rfq.vault_positions (ticket_id);

-- Find positions by PDA
CREATE INDEX idx_vault_positions_pda ON rfq.vault_positions (position_pda)
    WHERE position_pda IS NOT NULL;

-- Find positions by MM wallet (for MM dashboard)
CREATE INDEX idx_vault_positions_mm_wallet ON rfq.vault_positions (mm_wallet, created_at DESC)
    WHERE mm_wallet IS NOT NULL;

-- Find positions by user wallet
CREATE INDEX idx_vault_positions_user_wallet ON rfq.vault_positions (user_wallet, created_at DESC)
    WHERE user_wallet IS NOT NULL;

-- ============================================================
-- ESCROW JOBS (DEPRECATED - use vault_positions for new flow)
-- Retained for backward compatibility with existing escrow positions.
--
-- vault_positions replaces escrow_jobs: the status/error/retry_count columns
-- on vault_positions serve the same purpose as escrow_jobs for the vault flow.
-- No separate job queue table is needed - the vault-jobs service queries
-- vault_positions directly via the pending index.
-- ============================================================
CREATE TABLE rfq.escrow_jobs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rfq_id          UUID NOT NULL REFERENCES rfq.parlay_tickets(id),
    quote_id        UUID NOT NULL REFERENCES rfq.quotes(id),
    user_id         TEXT NOT NULL,
    mm_id           TEXT NOT NULL,
    status          TEXT NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'processing', 'completed', 'failed', 'cancelled')),
    attempt_count   INTEGER NOT NULL DEFAULT 0,
    max_attempts    INTEGER NOT NULL DEFAULT 5,
    last_error      TEXT,
    next_attempt_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_escrow_jobs_pending ON rfq.escrow_jobs (next_attempt_at)
    WHERE status IN ('pending', 'processing');
CREATE INDEX idx_escrow_jobs_rfq ON rfq.escrow_jobs (rfq_id);
CREATE UNIQUE INDEX idx_escrow_jobs_rfq_active ON rfq.escrow_jobs (rfq_id)
    WHERE status NOT IN ('failed', 'cancelled', 'completed');

    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rfq_id       UUID REFERENCES rfq.parlay_tickets(id) ON DELETE CASCADE,
    event_type   VARCHAR(50) NOT NULL
                 CHECK (event_type IN ('quote_accepted', 'position_created', 'position_settled', 'position_cancelled', 'position_failed')),

    user_wallet  VARCHAR(44),
    mm_wallet    VARCHAR(44),
    amount       NUMERIC(15, 2),
    block_time   TIMESTAMPTZ,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_blockchain_events_rfq ON rfq.blockchain_events (rfq_id);
CREATE INDEX idx_blockchain_events_tx  ON rfq.blockchain_events (tx_signature);
```

## Vault Position Status State Machine

```
                                ┌─────────────────┐
                                │     pending     │
                                │  (created in DB)│
                                └────────┬────────┘
                                         │
                    ┌────────────────────┼────────────────────┐
                    │                    │                    │
                    ▼                    ▼                    ▼
           ┌───────────────┐    ┌───────────────┐    ┌───────────────┐
           │    active     │    │  retry < MAX  │    │     error     │
           │  (on-chain)   │    │  (try again)  │    │ (MAX retries) │
           └───────┬───────┘    └───────┬───────┘    └───────────────┘
                   │                    │                  (terminal)
                   │                    └──► back to pending
        ┌──────────┴──────────┐
        │                     │
        ▼                     ▼
┌───────────────┐     ┌───────────────┐
│   settling    │     │  cancelling   │
│ (claimed by   │     │ (claimed by   │
│   worker)     │     │   worker)     │
└───────┬───────┘     └───────┬───────┘
        │                     │
   ┌────┴────┐           ┌────┴────┐
   │         │           │         │
   ▼         ▼           ▼         ▼
┌──────┐  ┌──────┐    ┌──────┐  ┌──────┐
│win   │  │loss  │    │cancel│  │revert│
│      │  │      │    │led   │  │to    │
│      │  │      │    │      │  │active│
└──────┘  └──────┘    └──────┘  └──────┘
```

**Retry exhaustion:** When `retry_count >= MAX_RETRIES` (default 5, configurable via
`VAULT_JOB_MAX_RETRIES` env var), the position transitions from `pending` to `error`.
A `position:error` WebSocket event is emitted. Manual intervention required to resolve.

## Scale Considerations

Designed for 200k DAU. Key operational notes:

### Table partitioning (implement before launch)


| Table            | Partition by | Strategy                                             |
| ---------------- | ------------ | ---------------------------------------------------- |
| `parlay_tickets` | `created_at` | Monthly - at 200k DAU = ~200k rows/day, ~73M/year    |
| `vault_positions`| `created_at` | Monthly - same scale as tickets                      |
| `rfq_events`     | `created_at` | Monthly - ~8 status changes per RFQ = ~1.6M rows/day |
| `user_pnl_daily` | `date`       | Yearly - 200k users × 365 days                       |


### Retention policies


| Table               | Retention                  | Reason                                                     |
| ------------------- | -------------------------- | ---------------------------------------------------------- |
| `market_sync_log`   | 7 days                     | High-frequency write (~1 row/min/series), operational only |
| `rfq_events`        | 90 days hot → cold storage | Compliance + support; cold = S3/ClickHouse after 90d       |
| `blockchain_events` | Indefinite                 | Immutable on-chain record                                  |


### Read routing

- `GET /markets`, `GET /leaderboard`, `GET /rfqs` (settled history) → **read replica**
- All writes, active RFQ queries, vault position state → **primary**

### Vault position queries

```sql
-- Find pending positions for job worker (single-instance safe)
SELECT * FROM rfq.vault_positions
WHERE status = 'pending' AND retry_count < 5
ORDER BY created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- Find active positions for settlement
SELECT vp.*, pt.user_id, pt.accepted_quote_id
FROM rfq.vault_positions vp
JOIN rfq.parlay_tickets pt ON pt.id = vp.ticket_id
WHERE vp.status = 'active'
ORDER BY vp.created_at ASC
LIMIT 50;

-- MM dashboard positions
SELECT * FROM rfq.vault_positions
WHERE mm_wallet = $1 AND status = 'active'
ORDER BY created_at DESC;
```
