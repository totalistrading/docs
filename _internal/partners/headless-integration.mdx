<Warning>
  This is an integration preview. The endpoints below are stable in design but not yet deployed. Use this page to plan your integration, and contact us for early access and a test environment.
</Warning>

In the headless model you hold the signing keys. For every on-chain action, Totalis builds the transaction and returns it unsigned, you sign it, and Totalis broadcasts and settles it. There is no delegation step and no browser interaction at any point.

- Base URL: `https://api.totalis.trade`
- Authentication: an API key in the `X-API-Key` header. Required scopes: `accounts:provision`, `trading:write`, `tx:sign`, `funds:withdraw`, `account:read`.

## Who It's For

- **Individuals** holding their own Solana keypair.
- **Platforms that custody their own users**, such as products built on Privy, Turnkey, or Fireblocks. You stay the custodian of record for your users and sign on their behalf. One Totalis API principal manages any number of sub-accounts, each with its own vault.

Signing is provider agnostic. Totalis verifies the signature, and how you produce it is up to you.

## How It Works

<Steps>
  <Step title="Register an account">
    Prove control of a public key, then register it. Each key becomes an account with its own vault.
  </Step>
  <Step title="Fund the account">
    Send USDC on-chain to the account's wallet address.
  </Step>
  <Step title="Quote and commit">
    Open a quote request, choose a quote, and commit, scoped to the sub-account.
  </Step>
  <Step title="Build, sign, submit">
    Totalis returns an unsigned transaction. You sign it and submit it back to broadcast.
  </Step>
  <Step title="Settle and withdraw">
    Positions settle server-side with no signature from you. Withdrawals use the same build, sign, and submit flow.
  </Step>
</Steps>

## 1. Register an Account

Registration is a two-step proof of control. Request a challenge, sign it, and submit the signature.

```bash
curl -X POST https://api.totalis.trade/v1/accounts/challenge \
  -H "X-API-Key: <key>" -H "Content-Type: application/json" \
  -d '{ "owner_pubkey": "<solana-pubkey>" }'
# returns { challenge_id, nonce, expires_at, message_to_sign }
```

Sign `message_to_sign` exactly as returned. It is canonical, domain-separated bytes that must be signed verbatim, not re-encoded. Then register:

```bash
curl -X POST https://api.totalis.trade/v1/accounts \
  -H "X-API-Key: <key>" -H "Content-Type: application/json" \
  -d '{ "owner_pubkey": "<pubkey>", "challenge_id": "<id>", "control_proof": "<signature>", "label": "your-user-ref" }'
# returns { id, owner_pubkey, vault_address }
```

Platforms repeat this per end-user. Each `owner_pubkey` becomes a sub-account with its own vault. Use `label` to record your own user identifier.

| Endpoint | Purpose |
| --- | --- |
| `POST /v1/accounts/challenge` | Request a single-use proof-of-control challenge |
| `POST /v1/accounts` | Register the account |
| `GET /v1/accounts`, `GET /v1/accounts/{id}` | List or read accounts |
| `DELETE /v1/accounts/{id}` | Disable an account. Funds remain withdrawable |

## 2. Fund

Send native USDC on Solana mainnet to the account's wallet address. There is no deposit endpoint. Placing a trade moves the stake into the vault automatically. No SOL is required, since gas is sponsored. The USDC mint is `EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v` (6 decimals).

## 3. Quote and Commit

Quoting follows the standard RFQ flow with one addition. Pass `X-Subaccount: <account_id>` on both create and commit so the position is attributed to the correct account.

```bash
# Open the quote request
curl -X POST https://api.totalis.trade/v1/quote-requests \
  -H "X-API-Key: <key>" -H "X-Subaccount: <account_id>" -H "Content-Type: application/json" \
  -d '{ "legs": [ {"market_ticker":"KXBTC-25FEB07-T100000","side":"yes","venue":"kalshi"} ], "bet_amount": 25 }'

# Fetch quotes and choose one
curl https://api.totalis.trade/v1/quote-requests/<quote_request_id>/quotes \
  -H "X-API-Key: <key>" -H "X-Subaccount: <account_id>"

# Commit the chosen quote
curl -X POST https://api.totalis.trade/v1/quote-requests/<quote_request_id>/commit \
  -H "X-API-Key: <key>" -H "X-Subaccount: <account_id>" -H "Content-Type: application/json" \
  -d '{ "quote_id": "<chosen-quote-id>" }'
```

Committing yields a `quote_request_id` and `quote_id`, both required by `build-tx`. In the headless model, committing does not create the position outright. It prepares the position for signing.

## 4. Build, Sign, Submit

Build the transaction, sign it with the account's key, and submit it back.

```bash
# Build returns an unsigned transaction
curl -X POST https://api.totalis.trade/v1/positions/build-tx \
  -H "X-API-Key: <key>" -H "X-Subaccount: <account_id>" -H "Content-Type: application/json" \
  -d '{ "quote_request_id": "<id>", "quote_id": "<id>" }'
# returns { build_id, transaction }

# Sign the transaction's message bytes, then submit
curl -X POST https://api.totalis.trade/v1/wallet/submit-built-tx \
  -H "X-API-Key: <key>" -H "Content-Type: application/json" \
  -d '{ "build_id": "<build_id>", "signed_transaction": "<base64>" }'
# returns { signature }
```

<Warning>
  Three constraints govern this flow:

  - The transaction from `build-tx` is not broadcastable on its own. Totalis adds its own signatures inside `submit-built-tx`, after verifying the submitted transaction matches what it built. Broadcasting it directly will fail.
  - Build, sign, and submit must run in sequence per `build_id`. Re-building while a signature is in flight replaces the cached transaction, and the earlier signature no longer matches.
  - Sign within about 60 to 90 seconds. The transaction carries a recent blockhash. If your signing round-trip is slower, request a new build and sign again.
</Warning>

## 5. Settlement

Positions settle server-side at market resolution, with no signature from the account. Proceeds land in the account's vault.

## 6. Withdraw

Withdrawals use the same build, sign, and submit flow. Funds return to the account's wallet.

```bash
curl -X POST https://api.totalis.trade/v1/vault/build-withdraw-tx \
  -H "X-API-Key: <key>" -H "X-Subaccount: <account_id>" -H "Content-Type: application/json" \
  -d '{ "amount_usdc": 50 }'
# returns { build_id, transaction }, then sign and POST /v1/wallet/submit-built-tx
```

## Reads

`GET /v1/wallet`, `GET /v1/vault`, and `GET /v1/portfolio` accept `X-Subaccount: <account_id>` to scope the response to a single account.
